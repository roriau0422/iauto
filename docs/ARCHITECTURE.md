# iAuto — System Architecture

**Status:** phases 1–5 backend complete; mobile (Expo) deferred
**Date:** 2026-04-02 (last updated 2026-05-03)
**Backend:** FastAPI (async)
**Mobile:** React Native via Expo (dev client / bare-adjacent) — deferred
**Scope source:** `docs/iAuto.docx.pdf`

---

## 1. Product in one paragraph

iAuto is a super-app for Mongolian drivers and auto businesses. A driver registers their vehicles via a plate-number lookup against the Mongolian government vehicle registry (exposed through the smartcar.mn XYP gateway; the lookup is executed by the mobile app on-device, not by the iAuto backend — see §3.7), searches for parts (text or image), chats with suppliers, reserves items for 24 hours, pays taxes/insurance/fines via QPay, logs service history, and — the two headline new features — gets AI-driven mechanical diagnosis and a data-driven market valuation of their car. Businesses manage inventory (warehouse), post stories, run targeted paid ads, and compete on ratings. The long-term moat is a data flywheel: every diagnosis, repair, sale, and warehouse event feeds the valuation and diagnostic models, which are impossible to replicate without owning the same ecosystem.

This is a marketplace + social + fintech + ML product in one. The architecture must reflect that it is **not a CRUD app**.

---

## 2. Guiding principles

1. **Modular monolith first, micro-services never (until forced).** One FastAPI deployable, clean bounded contexts inside. Split only when organizational or scaling pressure is real.
2. **Deterministic spine, AI as tools around it.** Marketplace, warehouse, payments, and ledgers are transactional and source-of-truth. AI features call into them via agent tools and never become sources of truth themselves.
3. **Data flywheel is a first-class architectural concern.** Every domain mutation emits a canonical event through an outbox. Events are the fuel for valuation and diagnostic models. The shape is in place on day one, even when volume is tiny.
4. **Contract-first.** OpenAPI is generated by FastAPI and consumed by the mobile app via type-safe codegen. Mobile and backend cannot drift.
5. **Async everywhere on the backend.** Chat, AI streaming, and background jobs will otherwise starve each other.
6. **Offline-tolerant mobile.** Mongolian mobile connectivity is variable. The app must be browsable cold and sync eventually.
7. **Tenant isolation at the repository layer.** Every query that touches business data filters by `tenant_id` explicitly. No Postgres row-level security — it is harder to debug than it is worth at this stage.
8. **Ruthless scope sequencing.** Marketplace first, AI later. The flywheel needs transactions to feed on.
9. **Team-size assumption: 4–6 engineers** (1 backend lead, 1 mobile lead, 1 ML, 1–3 full-stack). At 2 engineers, phase 3 and 4 timelines roughly double. This plan is fiction under that assumption.

## 3. Backend — FastAPI modular monolith

### 3.1 Bounded contexts

One FastAPI app, one deployable, organized by context. No cross-context direct database reads; each context exposes a service-layer API that others call in-process.

```
backend/
  app/
    identity/       # OTP, sessions, JWT+refresh, device registry, RBAC
    vehicles/       # registration via client-side XYP plan, service history, technical passport
    catalog/        # country→brand→model taxonomy, part categories
    marketplace/    # part-search requests, quotes, 24h reservations, sales, ratings
    chat/           # 1:1 messaging, presence, attachments
    stories/        # iAuto Story feed, ephemeral 12h TTL, premium contracts
    ads/            # paid promotion, targeting, billing, 48h reports
    warehouse/      # business inventory, bulk ops, reports, PDF/Excel export
    payments/       # QPay integration, invoicing, tax/insurance/fines, refunds
    ai_mechanic/    # diagnostic agent, multi-modal input, RAG
    valuation/      # car valuation ML inference + feature store reads
    notifications/  # push (FCM/APNs), in-app, email
    moderation/     # content review queue, user reports, admin tools
    analytics/      # event store, rollups, ML feature pipelines (write-side)
    admin/          # SQLAdmin mount — CMS + moderation tools
    platform/       # shared: db, cache, auth deps, outbox, telemetry, config
```

Each context has: `router.py` (HTTP), `service.py` (business logic), `repository.py` (DB), `models.py` (SQLAlchemy), `schemas.py` (Pydantic), `events.py` (outbox event shapes).

### 3.2 Data store

- **Postgres 16** as the single primary store. Extensions enabled: `pgvector` (embeddings), `pg_trgm` (fuzzy search on parts), `uuid-ossp`, `citext`.
- **Redis 7** for: OTP cache, rate limiting, session revocation list, Arq broker, presence, story TTL index, embedding cache.
- **S3-compatible object storage** for media: images, audio, PDFs, Excel exports, model artifacts. Presigned URLs; never proxy bytes through FastAPI.

One store, one backup surface, one monitoring story until traffic forces otherwise.

### 3.3 Async runtime

- FastAPI async handlers.
- SQLAlchemy 2.0 async + asyncpg driver.
- Pydantic v2 for schemas.
- **Arq** for background jobs (Redis-backed, async, fits FastAPI naturally). Handles: reservation-expiry, 48h ad reports, notification fanout, scraping pipelines, ML batch scoring, outbox consumers.

### 3.4 Outbox + event spine

Every context writes a row to the `outbox_events` table inside the same transaction as its domain mutation. An Arq consumer tails the outbox, dispatches events to in-process subscribers, and marks them dispatched. This gives exactly-once processing semantics on top of the DB transaction and no new infrastructure.

Canonical events (first cut):

```
identity.user_registered
vehicles.vehicle_registered
vehicles.service_logged
marketplace.part_search_submitted
marketplace.quote_sent
marketplace.reservation_started
marketplace.reservation_expired
marketplace.sale_completed
ads.ad_posted / ad_delivered / ad_report_generated
stories.story_published / story_expired
ai_mechanic.diagnosis_requested
ai_mechanic.diagnosis_followup_action
valuation.valuation_queried
valuation.valuation_converted_to_sale
payments.payment_settled / payment_failed
warehouse.item_added / item_sold / item_removed
```

Events are the raw fuel for analytics rollups and ML training. The same consumer writes them into a read-optimized `events_archive` table (JSONB payload + partitioned by month) for analytics queries. **Upgrade path:** when event volume or consumer diversity demands it, swap the Arq consumer for a dedicated message broker without changing the producer-side code.

### 3.5 Real-time messaging

- WebSocket connection per user, handled natively in FastAPI/Starlette.
- Redis Pub/Sub fan-out between worker processes — any worker can receive a message for a user connected to any other worker.
- Server-Sent Events for AI Mechanic streaming responses (simpler than WebSocket for uni-directional LLM token streams).
- Push notifications via FCM + APNs delivered by an Arq worker consuming notification events.

**Upgrade path:** if chat volume and presence state outgrow Redis Pub/Sub, introduce a dedicated real-time server. Not before.

### 3.6 Auth and tenancy

- Phone + OTP as the only identity factor. OTP delivery via SMS provider (operator choice; abstract behind a provider interface from day one). Phase 0 ships MessagePro as the sole rail.
- JWT access token (15 min) + refresh token (30 days, rotating). Refresh tokens stored hashed in DB with device binding; reuse of a revoked refresh token revokes the whole device chain.
- Multi-device allowed (spec 2.3). Device table tracks active sessions for revocation.
- Roles: `driver`, `business`, `admin`. The role can be selected at `POST /v1/auth/otp/verify` for **new** users only (`admin` is never self-assignable — rejected at the schema layer). Returning users keep their original role; promotion between `driver` and `business` is an admin-only operation.
- **Tenant model:** a user with `role='business'` owns exactly one `businesses` row (partial unique on `businesses.owner_id`). `businesses.id` is the `tenant_id` every tenant-scoped table will carry (warehouse items, stories, ads, reports, and the quotes side of the marketplace). A `business_members` pivot can add multi-staff later without a schema rewrite. `businesses.contact_phone` is an optional override encrypted with the same Fernet + HMAC blind-index envelope as `users.phone`.
- `get_current_business` FastAPI dependency resolves the authenticated user to their `Business` row — 403 for non-business roles, 404 when no profile exists yet. This is the gate every tenant-scoped endpoint uses.
- Every repository query that touches tenant-scoped tables takes `tenant_id` as a required argument. Enforced in code review and a lint-level check, not in the database.

### 3.7 Vehicle lookup — client-side smartcar.mn XYP gateway

Mongolia's government vehicle data is exposed through **smartcar.mn's public XYP gateway** (`https://xyp-api.smartcar.mn/xyp-api/v1/xyp/get-data-public`, service code `WS100401_getVehicleInfo`). The gateway expects a browser-impersonating header set from a legitimate smartcar.mn session — `Origin` / `Referer` of `https://smartcar.mn`, custom `os: web`, `version: 3.2.0`, and a Chrome `User-Agent`. A single iAuto backend IP calling this endpoint at scale is indistinguishable from a scraper and would be rate-limited or blocked within days.

**The backend never calls smartcar.mn or xyp.gov.mn.** The mobile app runs the lookup on-device — every request looks like an individual user's browser session to the gateway, fanned out across thousands of residential IPs. The server owns the *plan* the client executes, not the execution.

**Lookup plan** (`vehicle_lookup_plans` table; exactly one row active at a time via a partial unique index on `is_active=true`):

- `endpoint_url`, `endpoint_method`, `headers`, `body_template`, `slots`, `expected`, `ttl_seconds`
- `expected.fields` lists the shape of a successful flat-JSON response. The fields iAuto persists on the `vehicles` row:
  - `markName` → `vehicles.make` (resolved to `vehicle_brand_id` via `CatalogService`)
  - `modelName` → `vehicles.model` (resolved to `vehicle_model_id`)
  - `buildYear` → `vehicles.build_year` (tolerant int/float/string coercion — smartcar.mn flip-flops)
  - `cabinNumber` → `vehicles.vin` (encrypted + blind-indexed for dedup)
  - `motorNumber` → `vehicles.engine_number`
  - `colorName` → `vehicles.color`
  - `capacity` → `vehicles.capacity_cc`
  - `className` → `vehicles.class_code` (e.g. `"B"`)
  - `fuelType` → `vehicles.fuel_type` (Mongolian, e.g. `"Бензин"`)
  - `importDate` → `vehicles.import_month` (ISO datetime truncated to day-01 — year+month is the policy bucket, the day is paperwork noise)
  - `wheelPosition` → `vehicles.steering_side` enum: `"Зүүн"` → `LHD`, `"Баруун"` → `RHD`
- The last four are the session-5 business-coverage matching key (country/brand/model/year range + LHD/RHD). Unknown `wheelPosition` strings leave `steering_side` null rather than silently defaulting — the business coverage query excludes nulls from the LHD/RHD filter.
- `expected.error_signatures` ships classifier rules for non-2xx bodies. Each entry has a `match` (`status` + `body_contains_any`), a `category`, an `alert_operator` flag, and `client_message_mn` / `client_message_en`
- Rotating smartcar's reverse-engineered headers — which change under anti-bot pressure — is a DB update, not a mobile release

**Mobile client flow:**

1. `GET /v1/vehicles/lookup/plan` — public, `Cache-Control: public, max-age={ttl_seconds}`
2. Substitute the user's plate into `body_template.customFields.plateNumber`
3. `POST` to `endpoint.url` with the plan's headers
4. On HTTP 200 (flat JSON body): `POST /v1/vehicles` with the raw payload for dedup + registration
5. On non-2xx: match the response against `expected.error_signatures`
   - matching signature → show `client_message_mn` to the user and stop (e.g. a `HTTP 400` with `text/plain;charset=UTF-8` body containing `олдсонгүй` means "plate not in registry" — a user typo, not an outage, and must never page the operator)
   - no match → `POST /v1/vehicles/lookup/report` with the raw error; the backend pages the operator

**Trust model for the returned payload:** the server treats the client-posted XYP data as untrusted input. Physical cars are deduped by `vehicles.vin` via a partial unique index (`WHERE vin IS NOT NULL`), and ownership is many-to-many via a `vehicle_ownerships (user_id, vehicle_id)` composite-PK pivot — the same physical car can be registered by several drivers (family, dealer + driver, business + employee) without duplicating the `vehicles` row. Forgery mitigation is a future moderation pass on VIN collisions; no automated detection yet.

**Operator pager** (`app/vehicles/alerts.py`): a Redis `INCR` keyed by status bucket (`5xx`, `3xx`, exact `4xx`) with `TTL = XYP_ALERT_WINDOW_SECONDS` (default 900 s). The first failure in a fresh window fires a MessagePro SMS to `OPERATOR_PHONE`; subsequent failures within the window only bump the counter. SMS body is budgeted at **168 characters**, not 180 — MessagePro auto-appends a `" Navi market"` sender tag (12 chars) to every outbound message.

**Defence in depth:** `VehiclesService._is_user_input_error` re-runs the `олдсонгүй`-class match on the server before paging. A buggy client that misreports a not-found 400 as an outage still produces a `vehicle_lookup_reports` row and a `vehicles.lookup_failed` event for audit, but no SMS fires.

**Upgrade path:** if iAuto ever obtains legitimate direct XYP / ATYT credentials, a server-side executor slots in behind a `LookupExecutor` abstraction without breaking the mobile contract. The client still calls `GET /v1/vehicles/lookup/plan`; the plan returns `{"executor": "server"}`, and the client `POST`s the plate to a new `/v1/vehicles/lookup/execute` endpoint. The aggressive-cache / circuit-breaker / background-re-verification story that used to live in this section applies at that point — not before.

### 3.8 Payments — QPay

QPay is Mongolia's dominant payment rail. Integration:

- A single `payments` context owns all money flows: tax, insurance, fines, paid ads, premium reports, warehouse restore fees, story promotions.
- QPay invoice API called server-side; deep link or QR returned to mobile.
- Webhook receiver verifies HMAC signature, writes to a `payment_events` table, and emits `payments.payment_settled` — all other contexts react to the event, not to QPay directly.
- Double-entry ledger table (`ledger_entries`) tracks every movement. Auditable, reconcilable, and the only way to survive a dispute.

### 3.9 Observability and ops

- OpenTelemetry traces on every request and every outbox consumer.
- Prometheus metrics (request latency, outbox lag, DB pool saturation, LLM cost per request, model inference latency).
- Structured JSON logs shipped to a log store (Grafana Loki as the cheap default).
- Sentry for exceptions, both backend and mobile.
- AI cost dashboard: per-user daily spend, per-endpoint cost, alerts when a user's daily LLM spend crosses a threshold.

### 3.10 Admin panel

Mount **SQLAdmin** inside the FastAPI app for MVP CRUD and moderation workflows. It gives category management, "Knowledgeable Driver" content authoring, content review queue, user/business management, and moderation actions with near-zero build cost. When SQLAdmin's ceiling is hit, replace with a dedicated React admin app — not before.

## 4. Mobile — React Native (Expo, dev client)

### 4.1 Workflow choice

Use **Expo with a custom dev client** and config plugins from day one — not managed, not bare. The required feature set will blow through managed workflow limits (rich push notifications, QPay deep links, camera, audio recording, file exports, and eventually BLE for OBD-II). A dev-client build gives the ergonomics of Expo with the freedom to add native modules. Eject to bare only if BLE or another native requirement forces it; plan the evaluation point during phase 3, not as a surprise.

### 4.2 Stack

- **TypeScript strict mode** everywhere.
- **Expo Router** for file-based navigation.
- **TanStack Query** for server state. Every API call is a query or mutation, caching and invalidation are explicit. Persistence plugin + SQLite for offline reads.
- **Zustand** for local UI state. No Redux.
- **React Hook Form + Zod** for forms. Zod schemas are the shared contract with the backend for critical flows (registration, diagnosis input).
- **Generated API client** from the backend's OpenAPI spec via `openapi-typescript` — regenerated in CI, zero drift.
- **expo-image** for images, **expo-av** for audio recording and playback, **expo-camera** for scanning license plates and parts, **expo-file-system** for local caches.
- **react-native-maps** for fuel-station and service-point maps.
- **expo-notifications** for push.
- **react-native-sse** (or native EventSource polyfill) for AI Mechanic streaming.
- **WebSocket client** for chat.
- **react-native-ble-plx** as the planned OBD-II module (phase 5).

### 4.3 Offline behavior

- TanStack Query persistence writes server responses to SQLite.
- Parts catalog, user's own vehicle records, service history, and story drafts are readable offline.
- Write operations go through a mutation queue that retries when connectivity returns. Chat messages show a `pending` state until acknowledged.

### 4.4 Payments UX

QPay flow: backend creates an invoice, returns deep link + QR. Mobile opens the QPay app via deep link or presents the QR. A WebSocket or polling subscription on the invoice ID transitions the UI to success when the backend receives the webhook. No WebView-based form fills.

## 5. AI Mechanic

Design objective: give a Mongolian driver a diagnosis as good as walking into a competent Prius-literate shop, and then funnel them into the parts and service features of the app.

### 5.1 Input modalities — split correctly

The three input modes in the spec (text, image, audio) map to **three different ML pipelines** that must not be conflated:

**Text.** User's Mongolian text → embed → RAG retrieval from the knowledge base (pre-filtered by vehicle context) → LLM with tools → structured diagnosis JSON.

**Image.** Two distinct use cases require different tools:
- **Dashboard warning lights** are a finite enumerable set (~150 icons). Use a fine-tuned lightweight classifier (MobileNet/EfficientNet fine-tuned on an open dashboard-light dataset) hosted as a small inference endpoint. Fast, cheap, reliable. **Not** CLIP zero-shot — zero-shot loses on a closed vocabulary.
- **Open-vocabulary visual questions** ("what part is this?", "is this leak serious?", "where is the body damage?") use a multimodal LLM (OpenAI vision or equivalent) called through the agent. CLIP is reserved for similarity-based part lookup against the warehouse index.

**Audio.** Two distinct pipelines:
- **Spoken descriptions** of the problem → Whisper transcription → text pipeline.
- **Engine/mechanical sound recordings** (knocking, whining, abnormal idle) are *not* a Whisper problem — Whisper produces garbage on non-speech. Send these to a multimodal LLM that accepts audio (GPT-4o audio or equivalent) and describe the sound. A fine-tuned mechanical-sound classifier is a phase-5 improvement once enough labeled recordings accumulate through the platform.

The router at the front of the agent inspects the input type and dispatches to the correct sub-pipeline. This fork is architectural and must be named in code, not hidden inside a generic "process input" function.

### 5.2 Agent orchestration

OpenAI Agents SDK as the orchestrator. The agent is given the following tools, each implemented as an in-process function inside the `ai_mechanic` context:

- `get_vehicle_context(vehicle_id)` — pulls make, model, year, engine, mileage, service history, recent repairs, ordered parts. Populated from the deterministic spine.
- `search_knowledge_base(query, make, model, year)` — pgvector retrieval with metadata pre-filter.
- `search_parts(make, model, part_name)` — hits the marketplace's warehouse index. Grounds the cost estimate in real platform prices.
- `estimate_labor_cost(service_type, region)` — averages from booked services.
- `ask_followup(question)` — returns an interactive turn to the user when the diagnosis is ambiguous.

Response is forced into a Pydantic schema via the Agents SDK `response_format`:

```
{
  "diagnoses": [
    {
      "name": str,
      "probability": float,
      "urgency": "immediate" | "soon" | "monitor",
      "parts_needed": [str],
      "part_cost_mnt": {"min": int, "max": int},
      "labor_cost_mnt": {"min": int, "max": int},
      "explanation_mn": str
    }
  ],
  "followup_questions": [str],
  "next_actions": [{"label": str, "deeplink": str}]
}
```

The `next_actions` list wires diagnoses directly into "search parts" and "book service" flows in the app. This is the spec's "complete loop" (section 15.3).

### 5.3 Knowledge base and RAG

Corpus:
1. OBD-II codes translated to Mongolian (seed corpus).
2. Platform's own repair data: anonymized and aggregated `(make, model, symptom) → (diagnosis, parts, cost)` patterns.
3. Moderator-curated content (manuals, common issues) uploaded through the admin panel.
4. Facebook-group knowledge: only after legal review and explicit owner permission. Flag, do not assume.

Storage: pgvector table with metadata columns — `{source, make, model, year_min, year_max, language, embedding}`. Retrieval always pre-filters by vehicle context before similarity search, which cuts noise and improves precision dramatically.

### 5.4 AI cost controls — mandatory from day one

LLM spend is the biggest variable cost. Build these in from the start:

- **Model routing.** Triage with a cheap model (Haiku, GPT-4o-mini, or equivalent). Escalate to a premium model only when the diagnosis confidence is low, the case is complex, or the user is on a Pro plan.
- **Per-user daily rate limits.** Free tier gets N diagnoses/day, configurable. Pro tier gets a higher limit. Enforced in Redis.
- **Embedding cache.** Vehicle context embeddings and KB embeddings are cached by `(vehicle_id, content_hash)`. Do not re-embed the same context every session.
- **Spend metering.** Every LLM call is logged with a `cost_usd` estimate. Per-user and global daily spend dashboards. Alert when a single user exceeds a threshold (likely abuse or a bug).
- **Prompt cache where the provider supports it** (Claude prompt caching, OpenAI prompt caching) — static system prompt and RAG context become cached prefixes.

## 6. Car Valuation

Design objective: become the Kelley Blue Book of Mongolia. This is a classic supervised ML problem wrapped in a careful UX. The data sources and the Mongolia-specific feature engineering are the moat.

### 6.1 Training pipeline (offline)

- **Data ingestion**, scheduled weekly through Arq jobs:
  - АТҮТ registry deltas (authoritative make/model/year/engine/transmission/import date).
  - Scraped listings from Unegui.mn and Facebook Marketplace. **Legal review required before scraper deployment** — this is flagged as a go/no-go gate in phase 4.
  - Platform's own closed sales (highest-quality signal, weighted heavily).
  - Platform's own service history and parts orders (proxy for vehicle condition).
- **Feature engineering** captures Mongolia-specific signals the spec calls out:
  - `cold_climate_decay` — depreciation multiplier adjusted for UB cold exposure, hybrid battery degradation.
  - `import_policy_regime` — categorical by import-duty era; shifts whole-market prices.
  - `regional_fuel_demand` — Prius premium in cities, diesel premium rurally.
  - `odometer_fraud_risk` — mileage/age ratio vs. fleet distribution; flags implausible readings without silently trusting them.
- **Model.** CatBoost as the primary algorithm. It handles high-cardinality categoricals (trim, engine code) natively and beats XGBoost on this kind of tabular data without feature pre-engineering. Train on 2–3 years of scraped and closed-sale data, target 50k–100k records in year one.
- **Experiment tracking.** MLflow, local — lightweight.
- **Model registry.** Trained artifacts written to S3 under a versioned path. A small manifest in Postgres tracks the active version.

### 6.2 Serving (online)

- Inference is served by the same FastAPI process via a `valuation` router. Loading a CatBoost model into memory at startup is cheap; no separate model server needed at this scale.
- Input: `vehicle_id`. Output: `{range_low, range_mid, range_high, trend_arrow, comparable_sales, confidence_score, red_flags}`.
- Latency budget: p95 < 500 ms. Precompute on vehicle registration; refresh on relevant events (new comparable sale, ATUT data change, monthly scheduled refresh).
- Hot-swap of model versions through a config flag and a registry lookup — no redeploy.
- **Feature store:** a `vehicle_features` Postgres table refreshed by a nightly Arq job, denormalized for inference. Graduates to a dedicated feature store only if team complexity justifies it.

### 6.3 Evaluation

- Shadow-deploy new models and log their predictions alongside the active model.
- Ground truth: closed in-platform sales.
- Metric: MAE in MNT and MAPE. Per-segment (make, year-bucket) breakdowns.
- Promote new models only after outperforming the active model on shadow traffic for two weeks.

## 7. Data flywheel

This is the moat. The architecture must make it trivially easy to capture, store, and use behavioral data.

- **Event store.** Every outbox consumer writes to `events_archive`, partitioned by month, JSONB payload, indexed by `(event_type, user_id, vehicle_id, occurred_at)`.
- **Rollups.** Nightly Arq jobs build analytics rollup tables: `daily_active_users`, `part_search_conversion`, `diagnosis_feedback_distribution`, `valuation_vs_actual_sale`. Live in the same Postgres.
- **ML feature pipelines** read from `events_archive` and rollups, never from OLTP tables. Clean boundary between transactional and analytical data.
- **Analytics warehouse upgrade path.** When event volume exceeds what Postgres can chew through, migrate `events_archive` to a dedicated OLAP store. Not a day-one concern.

Critical events to instrument from day one, even if no one reads them yet:

```
part_search_submitted → part_search_matched → quote_sent → quote_accepted → sale_completed
diagnosis_requested → diagnosis_delivered → diagnosis_acted_on → repair_confirmed
valuation_queried → listing_posted → sale_completed_in_platform
warehouse_item_added → warehouse_item_sold (with price and margin)
```

Each event feeds a specific model's training data. Instrumenting them late means the training set starts late.

## 8. Privacy, compliance, and trust

- **Mongolia's Personal Information Protection Law (2022).** Treat it as GDPR-lite. Explicit consent on registration, data deletion on request, audit log for any admin read of PII.
- **Column-level encryption** on phone number, VIN, and license plate. Decrypted only in the service layer when explicitly needed.
- **PII redaction in logs.** Filter at the logging layer, not at each log site.
- **Scraped data legal review.** Unegui.mn and Facebook Marketplace scraping is the biggest legal risk. Do not deploy scrapers without written legal sign-off; be prepared to pay for licensed data as a fallback.
- **Moderation.** Queue, admin UI (SQLAdmin extension), user-report flow, hide/delete/ban actions. Stories, chat attachments, reviews, and paid ads all flow through moderation.

## 9. Repo and dev environment

Monorepo:

```
iauto/
  backend/          # FastAPI app, tests, Dockerfile
  mobile/           # Expo app
  ml/               # training pipelines, notebooks, CatBoost experiments
  shared/           # OpenAPI snapshot, shared contracts, codegen output
  infra/            # docker-compose, Terraform (later), deploy scripts
  docs/             # product docs and this file
```

- **Backend tooling:** `uv` for dependency and env management (fast, modern, replaces poetry+pyenv). `ruff` + `mypy` + `pytest` + `pytest-asyncio` + `testcontainers-postgres`. Pre-commit hooks.
- **Mobile tooling:** `pnpm` + `expo` + `eas`. Jest + React Testing Library for units, Maestro for E2E.
- **CI:** GitHub Actions. On every PR: backend lint + tests + OpenAPI diff, mobile type-check + tests + build. On main: Docker image build + push, Expo EAS build for mobile.
- **Dev environment:** `docker-compose` brings up Postgres, Redis, MinIO (S3), and the FastAPI app. Mobile runs against the local backend via the Expo dev client.
- **Database migrations:** **Alembic with a single migration history**, not per-context. The 14 bounded contexts share one Postgres; splitting migration history by context would turn every merge into a coordination problem. One `alembic/versions/` directory, one upgrade head, reviewed as part of the PR that introduces the schema change.
- **API versioning.** All routes are mounted under `/v1/`. The mobile app ships app-store builds that cannot be force-updated — old clients will call `/v1/` long after the server has moved on. Breaking changes introduce `/v2/` and run both in parallel for at least one full release cycle (minimum 90 days, measured from the day the `/v2/`-capable mobile build reaches production). A `Sunset` header communicates deprecation schedules to clients. This matters from phase 1 onward.

## 10. Deployment — start boring

Reality check: this team will not operate Kubernetes well in year one. Deploy boringly.

- **Production v1:** one cloud VM per role (API, worker, Postgres, Redis) or a managed Postgres + a container host (Render, Railway, Fly.io, or a regional equivalent). Caddy as the reverse proxy with automatic TLS. Automated nightly backups with PITR on Postgres.
- **Region:** Seoul or Tokyo for latency to Mongolia, unless a local provider is required for data residency.
- **Mobile distribution:** EAS Build + EAS Update for OTA JS updates. Published to Play Store and App Store through EAS Submit.

## 12. Risks and gates

- **smartcar.mn XYP gateway stability** — the mobile client absorbs transient failures per request; recurring outages are paged to the operator via the coalesced SMS alert (§3.7). If smartcar rotates their anti-bot header set, rotate the active row in `vehicle_lookup_plans` — no mobile release needed. If smartcar blocks wholesale, fall back to manual plate entry (`verification_source = 'manual'`) until a direct XYP credential is obtained.
- **Scraping legality** — phase 4 gate. If blocked, the valuation model starts with a smaller feature set and uses platform data as the primary signal.
- **LLM cost blowup** — cost controls are mandatory from day one of phase 3. Not optional.
- **Chat scale** — Redis Pub/Sub handles tens of thousands of concurrent users comfortably; revisit if approaching that number.
- **Moderation load** — user-generated content (stories, reviews, chat) scales with adoption. Plan for a dedicated moderator before launching stories broadly.
- **Data privacy incident** — tenant-scoping discipline at the repository layer plus PII encryption plus audit logging. A single leak of business warehouse data to a competitor would be unrecoverable.

## 13. The twelve decisions that matter most

1. Modular monolith in FastAPI, not microservices.
2. Postgres with pgvector as the only primary store. No FAISS, no Kafka, no ClickHouse day one.
3. Outbox + event store from day one. Data flywheel is non-negotiable.
4. AI as tools wrapping a deterministic marketplace spine. LLMs never own source-of-truth.
5. Dashboard lights are a classifier problem, not a CLIP similarity problem.
6. Engine sounds are not a Whisper problem — multimodal LLM or a specialized classifier.
7. Expo dev client, not managed. Plan native-module access from day one.
8. OpenAPI contract-first with generated mobile client. Zero drift. The live spec is served at `/openapi.json`; a committed snapshot at `shared/openapi/v1.json` is the mobile-codegen source and the CI drift-detection target. Regenerate the snapshot whenever a route or schema changes.
9. SQLAdmin as the admin panel until it breaks. Don't build a dashboard from scratch.
10. Repository-layer `tenant_id` filtering. No Postgres RLS.
11. Vehicle XYP lookups are client-side. The mobile app executes the smartcar.mn call on-device using a versioned server-controlled plan; the backend never holds an HTTP client to the gateway. Physical cars dedup by VIN via a partial unique index; ownership is many-to-many via `vehicle_ownerships`. Operator alerts are Redis-coalesced and paged via MessagePro (168-char body budget accounting for the auto-appended `" Navi market"` sender suffix).
12. AI cost controls (rate limit, model routing, embedding cache, spend alerts) are built into phase 3, not bolted on after a billing surprise.
