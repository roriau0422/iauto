# iAuto — Системийн архитектур

**Төлөв:** анхны төлөвлөгөө, хөгжүүлэлт эхлээгүй
**Огноо:** 2026-04-02
**Backend:** FastAPI (async)
**Мобайл:** React Native, Expo (dev client / bare-adjacent)
**Эх сурвалж:** `docs/iAuto.docx.pdf`

---

## 1. Бүтээгдэхүүний товч тайлбар

iAuto бол Монголын жолооч нар болон автомашины бизнесүүдэд зориулсан супер апп юм. Жолооч өөрийн тээврийн хэрэгслийг АТҮТ-ээр бүртгүүлж, сэлбэг хайж (текст эсвэл зургаар), нийлүүлэгчидтэй чатлаж, бараа 24 цагаар захиалж, татвар/даатгал/торгуулийг QPay-ээр төлж, засварын түүхээ хөтөлнө. Хамгийн гол шинэ хоёр боломж нь: AI-д суурилсан механикийн оношлогоо, мөн дата-д суурилсан машины зах зээлийн үнэлгээ. Бизнесүүд агуулахаа удирдаж, story нийтэлж, зорилтот сурталчилгаа явуулж, рейтингээр өрсөлдөнө. Урт хугацааны давуу тал нь дата-н flywheel: оношлогоо, засвар, худалдаа, агуулахын үйл явц бүр үнэлгээ болон оношлогооны загварыг тэжээнэ — ижил экосистемгүйгээр хуулбарлах боломжгүй.

Энэ бол marketplace + social + fintech + ML бүтээгдэхүүн нэг дор. Архитектур нь ердийн CRUD апп **биш** гэдгийг тусгасан байх ёстой.

## 2. Удирдлагын зарчмууд

1. **Модулиар монолит эхэлж, /micro-service/ хэзээ ч үгүй (шаардлагатай болтол).** Нэг FastAPI deployable, дотроо цэвэр bounded context-үүд. Зөвхөн байгууллагын бүтэц эсвэл ачааллын шаардлага бодит болсон үед л салга.
2. **Детерминистик суурь, AI нь хэрэгсэл.** Marketplace, агуулах, төлбөр, дансны бүртгэл — эдгээр нь транзакшны эх үүсвэр. AI функцүүд эдгээрийг agent tool-аар дамжуулан дуудна, хэзээ ч эх үүсвэр болохгүй.
3. **Дата flywheel нь архитектурын нэгдүгээр зэрэглэлийн шаардлага.** Домайны өөрчлөлт бүр outbox-оор дамжуулан гол event үүсгэнэ. Event-үүд нь үнэлгээ, оношлогооны загваруудын гол хүчиг зүйл. Хэлбэр нь эхний өдрөөс бэлэн байх ёстой, хэдий өгөгдөл бага байсан ч.
4. **Гэрээ түрүүлнэ (contract-first).** OpenAPI-г FastAPI үүсгэж, мобайл апп type-safe codegen-ээр хэрэглэнэ. Мобайл ба backend зөрөхийг зөвшөөрөхгүй.
5. **Backend дээр бүгд async.** Chat, AI streaming, background ажлууд бие биенээ гацаана.
6. **Мобайл offline-д тэсвэртэй.** Монголын мобайл холболт тогтворгүй. Апп холболтгүй үед ч үзэж болох, дараа нь sync хийх ёстой.
7. **Tenant тусгаарлалт repository давхарга дээр.** Бизнесийн өгөгдөлд хандсан query бүр `tenant_id`-ээр шүүнэ. Postgres RLS ашиглахгүй — энэ үе шатанд debug хийхэд хэцүү.
8. **Хамрах хүрээг хатуу дарааллаар.** Marketplace эхэлж, AI дараа нь. Flywheel-д тэжээх транзакш хэрэгтэй.

## 3. Backend — FastAPI модулиар монолит

### 3.1 Bounded context-үүд

Нэг FastAPI апп, нэг deployable, context-ээр зохион байгуулагдсан. Context хооронд шууд мэдээллийн сантай холбогдохгүй; context бүр service-layer API-аар дамжуулан бусдад хандана.

```
backend/
  app/
    identity/       # OTP, session, JWT+refresh, төхөөрөмжийн бүртгэл, RBAC
    vehicles/       # бүртгэл, АТҮТ интеграци, засварын түүх, техникийн паспорт
    catalog/        # улс→брэнд→загвар таксономи, сэлбэгийн ангилал
    marketplace/    # сэлбэг хайлт, үнийн санал, 24 цагийн захиалга, борлуулалт, үнэлгээ
    chat/           # 1:1 мессеж, presence, хавсралт
    stories/        # iAuto Story фийд, 12 цагийн TTL, premium гэрээ
    ads/            # төлбөртэй сурталчилгаа, зорилтот, тооцоо, 48 цагийн тайлан
    warehouse/      # бизнесийн бараа материал, бөөнөөр хийх үйлдэл, тайлан, PDF/Excel экспорт
    payments/       # QPay интеграци, нэхэмжлэл, татвар/даатгал/торгууль, буцаалт
    ai_mechanic/    # оношлогооны agent, олон модаль оролт, RAG
    valuation/      # машины үнэлгээний ML inference + feature store унших
    notifications/  # push (FCM/APNs), апп доторх, имэйл
    moderation/     # контент хянах дараалал, хэрэглэгчийн мэдэгдэл, админ хэрэгсэл
    analytics/      # event store, rollup, ML feature pipeline (бичих тал)
    admin/          # SQLAdmin mount — CMS + модераци хэрэгсэл
    platform/       # хуваалцсан: db, cache, auth dep, outbox, telemetry, config
```

Context бүр: `router.py` (HTTP), `service.py` (бизнес логик), `repository.py` (DB), `models.py` (SQLAlchemy), `schemas.py` (Pydantic), `events.py` (outbox event хэлбэрүүд).

### 3.2 Өгөгдлийн сан

- **Postgres 16** — цорын ганц үндсэн сан. Идэвхжүүлсэн өргөтгөлүүд: `pgvector` (embedding), `pg_trgm` (сэлбэг дээр бүдэг хайлт), `uuid-ossp`, `citext`.
- **Redis 7**: OTP кэш, rate limiting, session хүчингүй болгох жагсаалт, Arq broker, presence, story TTL индекс, embedding кэш.
- **S3-тэй нийцтэй объект хадгалалт** медиа-д зориулсан: зураг, аудио, PDF, Excel экспорт, загварын artifact. Presigned URL; FastAPI-аар дамжуулж байт дамжуулахгүй.

Нэг сан, нэг нөөцлөлтийн гадаргуу, нэг мониторинг — траффик шаардах хүртэл.

### 3.3 Async runtime

- FastAPI async handler-ууд.
- SQLAlchemy 2.0 async + asyncpg драйвер.
- Pydantic v2 schema-д.
- **Arq** background ажлуудад (Redis-д суурилсан, async, FastAPI-тай байгалийн нийцтэй). Зохицуулах зүйлс: захиалга дуусгавар болгох, 48 цагийн сурталчилгааны тайлан, мэдэгдэл тараах, scraping pipeline, ML batch scoring, outbox consumer.

### 3.4 Outbox + event сүүр

Context бүр домайны өөрчлөлттэй ижил транзакш дотор `outbox_events` хүснэгтэд мөр бичнэ. Arq consumer outbox-г дагаж, event-үүдийг in-process subscriber-уудад дамжуулж, dispatched гэж тэмдэглэнэ. DB транзакш дээр exactly-once боловсруулалт авч, шинэ дэд бүтэц нэмэхгүй.

Канон event-үүд (анхны хувилбар):

```
identity.user_registered
vehicles.vehicle_registered
vehicles.service_logged
marketplace.part_search_submitted
marketplace.quote_sent
marketplace.reservation_started
marketplace.reservation_expired
marketplace.sale_completed
ads.ad_posted / ad_delivered / ad_report_generated
stories.story_published / story_expired
ai_mechanic.diagnosis_requested
ai_mechanic.diagnosis_followup_action
valuation.valuation_queried
valuation.valuation_converted_to_sale
payments.payment_settled / payment_failed
warehouse.item_added / item_sold / item_removed
```

Event-үүд нь аналитикийн rollup болон машин сургалтын түүхий эд. Ижил consumer `events_archive` хүснэгтэд (JSONB payload + сараар partition хийсэн) бичнэ. **Шинэчлэлийн зам:** event-ийн хэмжээ эсвэл consumer-ийн олон янз байдал шаардвал Arq consumer-г зориулалтын message broker-оор солих — producer талын кодыг өөрчлөхгүйгээр.

### 3.5 Бодит цагийн мессеж

- Хэрэглэгч бүрд WebSocket холболт, FastAPI/Starlette дээр шууд зохицуулагдана.
- Redis Pub/Sub-ээр worker процессуудын хооронд fan-out — аль ч worker нь өөр worker-т холбогдсон хэрэглэгчийн мессежийг хүлээн авч чадна.
- AI Mechanic-ийн streaming хариултад Server-Sent Events (нэг чиглэлтэй LLM token stream-д WebSocket-ээс энгийн).
- FCM + APNs-ээр push мэдэгдэл, notification event-үүдийг хэрэглэсэн Arq worker-ээр дамжуулна.

**Шинэчлэлийн зам:** чат болон presence төлөв Redis Pub/Sub-ийн хүчин чадлаас хэтэрвэл зориулалтын бодит цагийн серверийг нэвтрүүлнэ. Түүнээс өмнө хийхгүй.

### 3.6 Нэвтрэлт ба tenant

- Утасны дугаар + OTP — цорын ганц таних хүчин зүйл. OTP-г SMS-ээр дамжуулна (оператор сонголт; эхний өдрөөс provider interface-ийн ард абстрактлана).
- JWT access token (15 мин) + refresh token (30 хоног, эргэлддэг). Refresh token-г DB-д hash-лаж, төхөөрөмжтэй холбож хадгална.
- Олон төхөөрөмж зөвшөөрнө (тодорхойлолт 2.3). Төхөөрөмжийн хүснэгт идэвхтэй session-уудыг хүчингүй болгоход хянана.
- Үүргүүд: `driver`, `business`, `admin`. Бизнес бүртгэл нь `tenant_id`-тэй бөгөөд агуулах, story, сурталчилгаа, тайланг хамрана.
- Tenant хамрах хүрээний хүснэгтэд хандсан repository query бүр `tenant_id`-г заавал авна. Код review болон lint түвшний шалгалтаар хяналт тавина, мэдээллийн санд биш.

### 3.7 АТҮТ (төрийн бүртгэл) интеграци — тогтвортой байдал заавал

Төрийн API найдваргүй. Тээврийн хэрэгсэл бүртгүүлэх процесс АТҮТ унтарсан үед амжилтгүй болох ёсгүй.

- Идэвхтэй кэшлэлт (тээврийн хэрэгслийн бүртгэлийн өгөгдөл ховор өөрчлөгддөг — 24 цаг кэшлэх, хэрэглэгчийн тодорхой хүсэлтээр шинэчлэх).
- Circuit breaker, exponential backoff-тэй.
- Graceful degradation: АТҮТ холбогдох боломжгүй бол гар оруулга хүлээн авна (марк, загвар, он, VIN) мөн тээврийн хэрэгслийг `unverified` гэж тэмдэглэнэ. Arq ажил background-д баталгаажуулалтыг дахин оролдож, амжилттай болсон үед `verified` болгоно.
- Дуудлага бүрийг зориулалтын `atut_calls` хүснэгтэд бүртгэнэ — debug болон rate бюджет хянахад.

### 3.8 Төлбөр — QPay

QPay бол Монголын давамгай төлбөрийн суваг. Интеграци:

- Нэг `payments` context бүх мөнгөн урсгалыг эзэмшинэ: татвар, даатгал, торгууль, төлбөртэй сурталчилгаа, premium тайлан, агуулахын сэргээлтийн хураамж, story сурталчилгаа.
- QPay нэхэмжлэлийн API-г серверээс дуудна; deep link эсвэл QR-г мобайлд буцаана.
- Webhook хүлээн авагч HMAC гарын үсгийг шалгаж, `payment_events` хүснэгтэд бичиж, `payments.payment_settled` event үүсгэнэ — бусад context-үүд event-д хариу үйлдэл хийнэ, QPay-д шууд биш.
- Давхар бичлэгийн дансны хүснэгт (`ledger_entries`) хөдөлгөөн бүрийг бүртгэнэ. Аудит хийх, тулгах боломжтой, маргаан даван туулах цорын ганц арга.

### 3.9 Хяналт ба үйл ажиллагаа

- OpenTelemetry trace хүсэлт бүр болон outbox consumer бүр дээр.
- Prometheus метрик (хүсэлтийн хоцролт, outbox lag, DB pool ачаалал, хүсэлт бүрийн LLM зардал, загварын inference хоцролт).
- Бүтэцтэй JSON лог, лог санд илгээнэ (Grafana Loki хямд сонголт).
- Sentry — backend болон мобайл хоёуланд exception-д.
- AI зардлын dashboard: хэрэглэгч бүрийн өдрийн зардал, endpoint бүрийн зардал, хэрэглэгчийн өдрийн LLM зардал босго давсан үед alert.

### 3.10 Админ самбар

**SQLAdmin**-г FastAPI апп дотор mount хийнэ — MVP CRUD болон модерацийн workflow-д. Ангиллын удирдлага, "Мэдлэгтэй жолооч" контент зохиох, контент хянах дараалал, хэрэглэгч/бизнес удирдлага, модерацийн үйлдлүүдийг бараг тэг зардлаар өгнө. SQLAdmin-ий хязгаар тулвал зориулалтын React админ апп хийх — түүнээс өмнө биш.

## 4. Мобайл — React Native (Expo, dev client)

### 4.1 Workflow сонголт

Эхний өдрөөс **Expo custom dev client** ба config plugin ашиглана — managed биш, bare биш. Шаардлагатай функцүүд managed workflow-ийн хязгаарыг давна (баялаг push мэдэгдэл, QPay deep link, камер, аудио бичлэг, файл экспорт, цаашид OBD-II-д BLE). Dev-client build нь Expo-ийн тав тухыг native модулийн эрх чөлөөтэй хослуулна. BLE эсвэл өөр native шаардлага шахвал bare-руу шилжих; үнэлгээний цэгийг 3-р алхамд төлөвлөна, гэнэтийн зүйл болгохгүй.

### 4.2 Стек

- **TypeScript strict mode** бүх газар.
- **Expo Router** файлд суурилсан навигацид.
- **TanStack Query** серверийн төлөвт. API дуудлага бүр query эсвэл mutation, кэшлэлт ба invalidation тодорхой. Persistence plugin + SQLite offline уншилтад.
- **Zustand** локал UI төлөвт. Redux үгүй.
- **React Hook Form + Zod** маягтуудад. Zod schema нь backend-тэй хуваалцсан гэрээ (бүртгэл, оношлогооны оролт зэрэг чухал flow-д).
- Backend-ийн OpenAPI spec-ээс `openapi-typescript`-ээр **үүсгэсэн API client** — CI-д дахин үүсгэнэ, тэг зөрүү.
- **expo-image** зурагт, **expo-av** аудио бичлэг ба тоглуулалтад, **expo-camera** улсын дугаар ба сэлбэг скан хийхэд, **expo-file-system** локал кэшүүдэд.
- **react-native-maps** шатахуун, засварын цэгийн газрын зурагт.
- **expo-notifications** push мэдэгдэлд.
- **react-native-sse** (эсвэл native EventSource polyfill) AI Mechanic streaming-д.
- **WebSocket client** чатад.
- **react-native-ble-plx** — төлөвлөсөн OBD-II модуль (алхам 5).

### 4.3 Offline зан төлөв

- TanStack Query persistence серверийн хариултуудыг SQLite-д бичнэ.
- Сэлбэгийн каталог, хэрэглэгчийн тээврийн хэрэгслийн бүртгэл, засварын түүх, story ноорог — offline уншигдана.
- Бичих үйлдлүүд mutation дарааллаар дамжиж, холболт сэргэхэд дахин оролдоно. Чат мессежүүд баталгаажих хүртэл `pending` төлөвтэй харагдана.

### 4.4 Төлбөрийн UX

QPay flow: backend нэхэмжлэл үүсгэж, deep link + QR буцаана. Мобайл QPay апп-г deep link-ээр нээх эсвэл QR харуулна. Нэхэмжлэлийн ID дээр WebSocket эсвэл polling subscription — backend webhook хүлээн авсан үед UI амжилт руу шилжинэ. WebView-д суурилсан маягт бөглөлт байхгүй.

## 5. AI Mechanic

Зорилго: Монголын жолоочид чадвартай Prius-ийн мэдлэгтэй засварын газар очсонтой ижил чанартай оношлогоо өгч, дараа нь аппын сэлбэг, засварын функцүүд рүү чиглүүлэх.

### 5.1 Оролтын модальууд — зөв салга

Тодорхойлолтын гурван оролтын горим (текст, зураг, аудио) нь **гурван өөр ML pipeline** руу буулгагдана — хольж болохгүй:

**Текст.** Хэрэглэгчийн Монгол текст → embed → мэдлэгийн сангаас RAG хайлт (тээврийн хэрэгслийн контекстоор урьдчилан шүүсэн) → хэрэгслүүдтэй LLM → бүтэцтэй оношлогоо JSON.

**Зураг.** Хоёр ялгаатай хэрэглээ өөр хэрэгсэл шаардана:
- **Хянах самбарын анхааруулах гэрлүүд** нь тоо нь хязгаарлагдмал (~150 дүрс тэмдэг). Нээлттэй dashboard-light dataset дээр fine-tune хийсэн хөнгөн classifier (MobileNet/EfficientNet) ашиглах. Хурдан, хямд, найдвартай. CLIP zero-shot **биш** — хаалттай үгийн санд zero-shot алдагдана.
- **Нээлттэй үгийн сангийн харааны асуултууд** ("энэ ямар сэлбэг вэ?", "энэ гоожилт ноцтой юу?", "их биеийн гэмтэл хаана байна?") нь multimodal LLM (OpenAI vision эсвэл түүнтэй дүйцэхүйц) agent-аар дамжуулан дуудагдана. CLIP нь зөвхөн агуулахын индексийн эсрэг сэлбэгийн ижил төстэй хайлтад хэрэглэгдэнэ.

**Аудио.** Хоёр ялгаатай pipeline:
- **Аман тайлбар** — Whisper-ээр хуулбарлаж → текст pipeline руу.
- **Хөдөлгүүр/механик дууны бичлэг** (тогших, инрэх, хэвийн бус idle) нь Whisper-ийн асуудал *биш* — Whisper яриа бус аудио дээр хог үр дүн гаргана. Эдгээрийг аудио хүлээн авдаг multimodal LLM (GPT-4o audio эсвэл түүнтэй дүйцэхүйц) руу илгээж, дууг тодорхойлно. Fine-tune хийсэн механик дууны classifier нь алхам 5-ийн сайжруулалт — платформоор дамжуулан хангалттай шошготой бичлэг хуримтлагдсаны дараа.

Agent-ийн өмнөх router нь оролтын төрлийг шалгаж зөв дэд pipeline руу чиглүүлнэ. Энэ салаалалт нь архитектурын түвшинд бөгөөд код дээр нэрлэгдсэн байх ёстой, ерөнхий "process input" функцийн ард нуугдахгүй.

### 5.2 Agent-ийн зохион байгуулалт

OpenAI Agents SDK-г зохицуулагчаар ашиглана. Agent-д дараах хэрэгслүүдийг өгнө, тус бүр `ai_mechanic` context доторх in-process функц:

- `get_vehicle_context(vehicle_id)` — марк, загвар, он, хөдөлгүүр, гүйлт, засварын түүх, сүүлийн засвар, захиалсан сэлбэг татах. Детерминистик сүүрээс дүүргэнэ.
- `search_knowledge_base(query, make, model, year)` — pgvector хайлт metadata-аар урьдчилан шүүсэн.
- `search_parts(make, model, part_name)` — marketplace-ийн агуулахын индекс рүү хандана. Зардлын тооцоог платформын бодит үнэд суурилуулна.
- `estimate_labor_cost(service_type, region)` — захиалсан үйлчилгээнүүдийн дундаж.
- `ask_followup(question)` — оношлогоо тодорхойгүй үед хэрэглэгчид интерактив эргэлт буцаана.

Хариуг Agents SDK-ийн `response_format`-аар Pydantic schema руу албадна:

```
{
  "diagnoses": [
    {
      "name": str,
      "probability": float,
      "urgency": "immediate" | "soon" | "monitor",
      "parts_needed": [str],
      "part_cost_mnt": {"min": int, "max": int},
      "labor_cost_mnt": {"min": int, "max": int},
      "explanation_mn": str
    }
  ],
  "followup_questions": [str],
  "next_actions": [{"label": str, "deeplink": str}]
}
```

`next_actions` жагсаалт нь оношлогоог аппын "сэлбэг хайх" ба "засвар захиалах" flow руу шууд холбоно. Энэ бол тодорхойлолтын "бүрэн хаалт" (хэсэг 15.3).

### 5.3 Мэдлэгийн сан ба RAG

Корпус:
1. Монгол хэл рүү орчуулсан OBD-II кодууд (анхны корпус).
2. Платформын өөрийн засварын өгөгдөл: нэргүйжүүлсэн, нэгтгэсэн `(марк, загвар, шинж тэмдэг) → (оношлогоо, сэлбэг, зардал)` хэв маяг.
3. Модератор зохиосон контент (гарын авлага, түгээмэл асуудлууд) админ самбараар дамжуулан оруулсан.
4. Facebook бүлгийн мэдлэг: зөвхөн хуулийн хянан шалгалт ба эзэмшигчийн тодорхой зөвшөөрлийн дараа. Тэмдэглэх, таамаглахгүй.

Хадгалалт: pgvector хүснэгт metadata баганатай — `{source, make, model, year_min, year_max, language, embedding}`. Хайлт нь similarity хайлтын өмнө тээврийн хэрэгслийн контекстоор үргэлж урьдчилан шүүнэ — шуугиан буурч, нарийвчлал мэдэгдэхүйц сайжирна.

### 5.4 AI зардлын хяналт — эхний өдрөөс заавал

LLM зардал бол хамгийн том хувьсах зардал. Эхнээсээ суулга:

- **Загвар чиглүүлэлт.** Хямд загвараар (Haiku, GPT-4o-mini эсвэл түүнтэй дүйцэхүйц) ангилна. Оношлогооны итгэлцэл бага, кейс нарийн төвөгтэй эсвэл Pro хэрэглэгч үед л premium загвар руу дээшлүүлнэ.
- **Хэрэглэгч бүрийн өдрийн rate limit.** Үнэгүй түвшин нь өдөрт N оношлогоо, тохируулж болно. Pro түвшин илүү өндөр хязгаартай. Redis-д хэрэгжүүлнэ.
- **Embedding кэш.** Тээврийн хэрэгслийн контекст embedding болон KB embedding-ийг `(vehicle_id, content_hash)`-аар кэшлэнэ. Session бүрт ижил контекстыг дахин embed хийхгүй.
- **Зардлын хэмжилт.** LLM дуудлага бүрийг `cost_usd` тооцоолсон зардалтай бүртгэнэ. Хэрэглэгч тус бүрийн болон глобал өдрийн зардлын dashboard. Нэг хэрэглэгч босго давсан үед alert (хэтрүүлсэн хэрэглээ эсвэл bug-ийн шинж).
- **Провайдер дэмжсэн тохиолдолд prompt cache** (Claude prompt caching, OpenAI prompt caching) — статик system prompt ба RAG контекст кэшлэгдсэн prefix болно.

## 6. Машины үнэлгээ

Зорилго: Монголын Kelley Blue Book болох. Энэ бол анхааралтай UX-д боосон сонгодог supervised ML асуудал. Өгөгдлийн эх сурвалж болон Монголд зориулсан feature engineering нь давуу тал.

### 6.1 Сургалтын pipeline (offline)

- **Өгөгдөл оруулах**, долоо хоног бүр Arq ажлуудаар:
  - АТҮТ бүртгэлийн дельта (эрх бүхий марк/загвар/он/хөдөлгүүр/хурдны хайрцаг/импортын огноо).
  - Unegui.mn болон Facebook Marketplace-ээс scrape хийсэн зарууд. **Scraper ажиллуулахаас өмнө хуулийн хянан шалгалт шаардлагатай** — алхам 4-т go/no-go хаалга болж тэмдэглэгдсэн.
  - Платформын өөрийн хаагдсан борлуулалт (хамгийн чанартай дохио, жинг ихэсгэнэ).
  - Платформын засварын түүх ба сэлбэг захиалгууд (тээврийн хэрэгслийн нөхцөлийн proxy).
- **Feature engineering** — Монголд зориулсан дохиог авна:
  - `cold_climate_decay` — УБ-ын хүйтэн уур амьсгалын нөлөөнд тохируулсан элэгдлийн коэффициент, гибрид батарей доройтол.
  - `import_policy_regime` — импортын татварын үеийн ангилал; бүхэл зах зээлийн үнийг шилжүүлдэг.
  - `regional_fuel_demand` — хотод Prius-ийн premium, хөдөө дизелийн premium.
  - `odometer_fraud_risk` — гүйлт/насны харьцааг паркийн хуваарилалттай харьцуулна; итгэмшигтэй бус уншилтыг тэмдэглэнэ, чимээгүй итгэхгүй.
- **Загвар.** CatBoost — үндсэн алгоритм. Өндөр кардиналтай категорийн хувьсагчдыг (trim, хөдөлгүүрийн код) шууд зохицуулдаг бөгөөд ийм хүснэгтэн өгөгдөл дээр XGBoost-ыг feature-ийн урьдчилсан инженерчлэлгүйгээр давна. 2–3 жилийн scrape ба хаагдсан борлуулалтын өгөгдөл дээр сургах, эхний жилд 50k–100k бичлэг зорино.
- **Туршилтын хяналт.** MLflow, локал — хөнгөн.
- **Загварын бүртгэл.** Сургасан artifact-уудыг S3-д хувилбарласан замаар бичнэ. Postgres дахь жижиг manifest идэвхтэй хувилбарыг хянана.

### 6.2 Үйлчлэлт (online)

- Inference-г ижил FastAPI процесс `valuation` router-аар үйлчилнэ. CatBoost загварыг эхлүүлэхэд санах ойд ачаалах нь хямд; энэ хэмжээнд тусдаа model server шаардлагагүй.
- Оролт: `vehicle_id`. Гаралт: `{range_low, range_mid, range_high, trend_arrow, comparable_sales, confidence_score, red_flags}`.
- Хоцролтын бюджет: p95 < 500 мс. Тээврийн хэрэгсэл бүртгэхэд урьдчилан тооцоолох; холбогдох event-ийн үед шинэчлэх (шинэ ижил төстэй борлуулалт, АТҮТ өгөгдлийн өөрчлөлт, сарын төлөвлөгөөт шинэчлэл).
- Config flag ба registry lookup-аар загварын хувилбарыг идэвхтэй солих — redeploy шаардлагагүй.
- **Feature store:** шөнийн Arq ажлаар шинэчлэгдэх `vehicle_features` Postgres хүснэгт, inference-д зориулж denormalize хийсэн. Багийн нарийн төвөгтэй байдал шаардвал зориулалтын feature store руу шилжинэ.

### 6.3 Үнэлгээ

- Шинэ загваруудыг shadow-deploy хийж, идэвхтэй загвартай зэрэгцүүлэн таамаглалыг бүртгэнэ.
- Ground truth: платформ дотроо хаагдсан борлуулалт.
- Метрик: MNT-ээр MAE ба MAPE. Сегмент бүрийн (марк, оны бүлэг) задаргаа.
- Хоёр долоо хоногийн shadow traffic дээр идэвхтэй загвараас давсны дараа л шинэ загваруудыг дэвшүүлнэ.

## 7. Дата-н flywheel

Энэ бол давуу тал. Архитектур нь зан үйлийн өгөгдлийг цуглуулах, хадгалах, ашиглахад маш хялбар байлгах ёстой.

- **Event store.** Outbox consumer бүр `events_archive` руу бичнэ, сараар partition хийсэн, JSONB payload, `(event_type, user_id, vehicle_id, occurred_at)`-аар индекслэсэн.
- **Rollup.** Шөнийн Arq ажлууд аналитикийн rollup хүснэгтүүд бүтээнэ: `daily_active_users`, `part_search_conversion`, `diagnosis_feedback_distribution`, `valuation_vs_actual_sale`. Ижил Postgres дотор.
- **ML feature pipeline** нь `events_archive` ба rollup-аас уншина, OLTP хүснэгтүүдээс хэзээ ч уншихгүй. Транзакшны ба аналитикийн өгөгдлийн хооронд цэвэр хил.
- **Аналитикийн warehouse шинэчлэлийн зам.** Event-ийн хэмжээ Postgres-ийн боловсруулах чадвараас хэтэрвэл `events_archive`-г зориулалтын OLAP сан руу шилжүүлнэ. Эхний өдрийн асуудал биш.

Эхний өдрөөс хэмжих чухал event-үүд, хэн ч уншаагүй байсан ч:

```
part_search_submitted → part_search_matched → quote_sent → quote_accepted → sale_completed
diagnosis_requested → diagnosis_delivered → diagnosis_acted_on → repair_confirmed
valuation_queried → listing_posted → sale_completed_in_platform
warehouse_item_added → warehouse_item_sold (үнэ, ашигтай)
```

Event бүр тодорхой загварын сургалтын өгөгдлийг тэжээнэ. Хожуу хэмжвэл сургалтын багц хожуу эхэлнэ.

## 9. Repo ба хөгжүүлэлтийн орчин

Monorepo:

```
iauto/
  backend/          # FastAPI апп, тест, Dockerfile
  mobile/           # Expo апп
  ml/               # сургалтын pipeline, notebook, CatBoost туршилтууд
  shared/           # OpenAPI snapshot, хуваалцсан гэрээ, codegen гаралт
  infra/            # docker-compose, Terraform (хожим), deploy скриптүүд
  docs/             # бүтээгдэхүүний баримт бичиг ба энэ файл
```

- **Backend хэрэгслүүд:** `uv` хамаарал ба орчин удирдлагад (хурдан, орчин үеийн, poetry+pyenv-г орлоно). `ruff` + `mypy` + `pytest` + `pytest-asyncio` + `testcontainers-postgres`. Pre-commit hook-ууд.
- **Мобайл хэрэгслүүд:** `pnpm` + `expo` + `eas`. Jest + React Testing Library нэгж тестэд, Maestro E2E-д.
- **CI:** GitHub Actions. PR бүрт: backend lint + тест + OpenAPI diff, мобайл type-check + тест + build. Main дээр: Docker image build + push, мобайл Expo EAS build.
- **Хөгжүүлэлтийн орчин:** `docker-compose` Postgres, Redis, MinIO (S3), FastAPI апп-г ажиллуулна. Мобайл Expo dev client-ээр локал backend-тэй холбогдоно.
- **Мэдээллийн сангийн migration:** **Alembic нэг migration түүхтэй**, context бүрт салгахгүй. 14 bounded context нэг Postgres хуваалцна; migration түүхийг context-ээр салгавал merge бүр координацийн асуудал болно. Нэг `alembic/versions/` хавтас, нэг upgrade head, schema өөрчлөлт оруулсан PR-ийн хэсэг болж шалгагдана.
- **API хувилбаржуулалт.** Бүх route `/v1/` дор mount хийгдэнэ. Мобайл апп нь app store-ийн build илгээдэг бөгөөд албадан шинэчлэх боломжгүй — хуучин client-үүд сервер урагшилсны дараа ч удаан `/v1/` дуудна. Эвдэх өөрчлөлт `/v2/` нэвтрүүлж, хоёуланг нь дор хаяж нэг бүтэн release мөчлөг (production-д `/v2/` чадвартай мобайл build хүрснээс хойш хамгийн багадаа 90 хоног) зэрэгцүүлж ажиллуулна. `Sunset` header хуучирсан хуваарийг client-уудад мэдэгдэнэ. алхам 1-ээс эхлэн хамаатай.

## 10. Deploy — энгийнээр эхэл

Бодит байдлын шалгалт: энэ баг эхний жилд Kubernetes-ийг сайн ажиллуулж чадахгүй. Энгийнээр deploy хий.

- **Мобайл түгээлт:** EAS Build + EAS Update OTA JS шинэчлэлтэд. EAS Submit-ээр Play Store ба App Store-д нийтэлнэ.
- **Шинэчлэлийн зам:** багийн хэмжээ ба үйл ажиллагааны туршлага зөвтгөсөн үед Kubernetes руу шилжинэ. Түүнээс өмнө биш.

## 11. Үе шатчилсан замын зураг

4–6 инженер гэж таамаглав. Үе шат бүр 4–6 долоо хоног. 2 хүнтэй бол хугацааг хоёр дахин нэмэгдүүл.

### алхам 0 — суурь (1–4 дэх долоо хоног)
- Monorepo, CI, хөгжүүлэлтийн орчин.
- FastAPI skeleton identity + vehicles + catalog context-уудтай.
- Expo апп skeleton навигаци ба үүсгэсэн API client-тэй.
- OTP нэвтрэлт бүрэн.
- АТҮТ интеграци ба graceful fallback-тай тээврийн хэрэгслийн бүртгэл.
- Outbox ба event store суурь.

### алхам 1 — гол marketplace (5–10 дахь долоо хоног)
- Сэлбэг хайлт (текст + зураг).
- Үнийн санал, 24 цагийн захиалга, борлуулалт дуусгах.
- Үнэлгээ ба шүүмж.
- WebSocket + Redis Pub/Sub чат.
- Push мэдэгдэл.
- QPay интеграци ба дансны бүртгэл.
- Миний машин (татвар/даатгал/торгууль).
- Засварын түүх ба PDF экспорт.

### алхам 2 — бизнес хэрэгслүүд (11–14 дэх долоо хоног)
- Мобайл **болон** зориулалтын бизнес вэб апп дээр агуулахын удирдлага. Вэб гадаргуу нь тодорхойлолтын шаардлага (хэсэг 11.1: "Апп болон вебээр ажиллана"), nice-to-have биш — бизнесүүд админ самбараар бараа материалаа удирдахгүй. Мобайл аппаар үүсгэсэн ижил OpenAPI client хэрэглэсэн хамгийн бага **Vite + React + TanStack Query** SPA болгон бүтээ. Ижил reverse proxy-ийн ард статик asset-аар deploy хий.
- iAuto Story фийд premium-гэрээний логиктой.
- Зорилтот ба 48 цагийн тайлантай төлбөртэй сурталчилгааны систем.
- Модераци ба контентын админ самбар (SQLAdmin) — зөвхөн дотоод хэрэглээ, бизнес агуулахын вэб апп-аас ялгаатай.

### алхам 3 — AI Mechanic MVP (15–20 дахь долоо хоног)
- RAG мэдлэгийн сангийн эхлэл (OBD-II кодууд + анхны зохиосон контент).
- Хянах самбарын анхааруулах гэрлийн classifier (нээлттэй dataset дээр fine-tune хийсэн MobileNet).
- Аудио pipeline салгалт: яринд Whisper, хөдөлгүүрийн дуунд multimodal LLM.
- Agents SDK — тээврийн хэрэгслийн контекст, сэлбэг хайлт, хөдөлмөрийн зардал, дагалдах асуулт хэрэгслүүдтэй.
- SSE streaming UI.
- Зардлын хяналт ба хэрэглэгч бүрийн rate limit-ийг энэ фазын эхний өдрөөс амьд болго.

### алхам 4 — Машины үнэлгээ (18–24 дэх долоо хоног, алхам 3-тай давхцана)
- Scraping-ийн хуулийн хянан шалгалт — **хаалга**. Сөрөг бол зөвхөн АТҮТ + платформын өгөгдөлд түшиглэж хамрах хүрээг багасга.

- Монголд зориулсан хүчин зүйлүүдтэй feature engineering.
- CatBoost V1 хүрээний тооцоо (үнэгүй түвшин).
- Үнэлгээний мобайл UX.
- Premium тайлангийн төлбөрийн flow.
- Shadow deployment framework.

### алхам 5 — давуу талыг хурдасгах

- Message broker шинэчлэлийн үнэлгээ.

## 12. Эрсдэл ба хаалгууд

- **АТҮТ API-ийн тогтвортой байдал** — архитектурын тогтвортой байдал нь бууруулна гэхдээ арилгахгүй. Гэмтлийн давтамжийг хяна.
- **Scraping-ийн хууль зүй** — алхам 4-ийн хаалга. Хориглогдвол үнэлгээний загвар жижиг feature-ийн багцаар эхэлж, платформын өгөгдлийг үндсэн дохио болгоно.
- **LLM зардлын хэт өсөлт** — алхам 3-ын эхний өдрөөс зардлын хяналт заавал. Нэмэлт сонголт биш.
- **Чат ачаалал** — Redis Pub/Sub хэдэн арван мянган зэрэгцээ хэрэглэгчийг тайвнаар зохицуулна; тэр тоонд ойртох үед дахин хянана.
- **Модерацийн ачаалал** — хэрэглэгчийн үүсгэсэн контент (story, шүүмж, чат) өсөлттэй хамт нэмэгдэнэ. Story-г өргөн нэвтрүүлэхээс өмнө зориулалтын модератор төлөвлө.
- **Өгөгдлийн нууцлалын хэрэг** — repository давхаргын tenant хамрах хүрээний сахилга + PII шифрлэлт + аудит лог. Бизнесийн агуулахын өгөгдөл өрсөлдөгч рүү нэг удаа задарвал нөхөж болохгүй.

## 13. Хамгийн чухал арван хоёр шийдвэр

1. FastAPI дээр модулиар монолит, microservice биш.
2. pgvector-тэй Postgres — цорын ганц үндсэн сан. FAISS үгүй, Kafka үгүй, ClickHouse эхний өдөр үгүй.
3. Outbox + event store эхний өдрөөс. Дата-н flywheel хэлэлцэх зүйлгүй.
4. AI нь детерминистик marketplace сүүрийг ороосон хэрэгсэл. LLM хэзээ ч эх үүсвэр эзэмшихгүй.
5. Хянах самбарын гэрэл нь classifier асуудал, CLIP ижил төстэй байдлын асуудал биш.
6. Хөдөлгүүрийн дуу нь Whisper-ийн асуудал биш — multimodal LLM эсвэл мэргэжлийн classifier.
7. Expo dev client, managed биш. Эхний өдрөөс native-модулийн хандалтыг төлөвлө.
8. OpenAPI contract-first, үүсгэсэн мобайл client-тэй. Тэг зөрүү.
9. SQLAdmin — эвдрэх хүртэл админ самбар. Dashboard-г эхнээсээ бүтээхгүй.
10. Repository-давхаргын `tenant_id` шүүлтүүр. Postgres RLS үгүй.
11. АТҮТ дуудлагууд кэш-түрүүлсэн, circuit breaker ба гар оруулгын fallback-тай байх ёстой.
12. AI зардлын хяналт (rate limit, загвар чиглүүлэлт, embedding кэш, зардлын alert) алхам 3-д суулгана, төлбөрийн гэнэтийн мэдэгдлийн дараа залгахгүй.
